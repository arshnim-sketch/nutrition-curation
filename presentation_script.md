# 💊 AI 맞춤형 영양제 큐레이션 플랫폼 발표 자료

> **주제:** LLM과 React를 결합한 개인 맞춤형 임상 영양 큐레이션 서비스

---

## 💡 1. 프로젝트 개요 및 문제 정의 (어떤 문제를 해결하는가?)
- **문제 인식**: 현대인은 수많은 영양제 광고에 노출되지만, **"내 몸과 증상에 정말 필요한가?" "여러 제품을 섞어 먹었을 때 상호작용은 안전한가?"**에 대한 판단을 내리기 매우 어렵습니다.
- **현재 시장의 한계**: 
  - 단일 증상 기반의 단순 제품 추천 (마케팅 위주)
  - 영양소 과다 복용, 성분 간 흡수 방해(예: 칼슘과 철분)에 대한 종합 분석 부재
- **해결책**: 사용자 인적사항(나이, 성별) 및 겪고 있는 건강 목표(증상)를 입력받아 **임상 영양학적 기준**으로 최적의 영양제 조합을 찾아주고, **영양소 밸런스와 상호작용까지 검증**해주는 AI 전문 큐레이션 서비스를 개발했습니다.

---

## 🤖 2. 왜 LLM이 필요한가?
- **복잡한 다중 추론의 영역**: 수십 가지 영양제에 포함된 복합 성분들을 합산하여 한국인 영양소 섭취기준(KDRIs)과 대조하고, 성분 간의 시너지/독성/복용 타이밍을 종합 판단하는 것은 `if-else` 등단순 하드코딩 룰베이스 기반으로는 구현이 불가능에 가깝습니다.
- **맥락에 맞는 자연스러운 설명 생성 (XAI)**: 추천된 조합이 "왜 이 환자의 현재 증상에 적합한지", "복용 시 어떤 주의가 필요한지"를 약사처럼 구체적이고 전문적인 자연어로 풀어 설명할 수 있는 탁월한 능력을 갖추고 있습니다.
- **JSON 구조화 강제**: 최신 LLM(`gpt-4o`)의 JSON 모드를 활용하여 프론트엔드가 즉각적으로 화면을 렌더링할 수 있는 규격화되고 예측 가능한 데이터 포맷을 반환받습니다.

---

## 🏗 3. 시스템 아키텍처 및 설계
- **🎨 UI (커스텀 클라이언트)**: React + Vite + TypeScript 형태의 전용 웹 클라이언트 구현.
  - [UI 설계 요건 충족] 단순 대화형 인터페이스를 넘어, 데이터를 시각화하고 상호작용(체크박스 토글 등)을 제공하는 클라이언트.
  - 모바일 퍼스트(Mobile-first) 기반의 깔끔하고 트렌디한 디자인 시스템 적용.
- **⚙️ AI 시스템 설계 (Prompting & RAG 개념식 주입)**:
  - 역할 부여: "임상 영양사 및 약사 전문가"
  - RAG 활용: 외부에서 수집한 자체 제품 카탈로그(`products.json`) 정보를 Context로 컨텍스트화시켜 LLM에 안전하게 주입(Injection)하여, 오직 DB 내의 제품만 추천하도록 제한.
  - 엄격한 제어: 복합 영양제(멀티비타민 등)를 반드시 하위 개별 영양소 단위로 분해해서 분석하도록 프롬프트 레벨에서 강하게 제어(Rule-based Prompting).
- **🔄 자동화 및 데이터 파이프라인 연동**:
  - 앱 사용 데이터 로깅: **Google Apps Script Webhook**을 구축하고 외부 API 연동을 통해 사용자 쿼리 및 AI 큐레이션 결과를 **Google Sheets에 실시간 데이터 로깅 및 자동화** 파이프라인 구성.

## 👀 4. 주요 기능 (Key Highlights)
1. **2-Step 큐레이션 마법사**: 관심 부위를 먼저 선택하면 그에 맞는 증상만 노출시켜 사용자 인지 부하 감소.
2. **Interactive Nutrient Balance (실시간 연동 UI)**: 
   - AI가 계산해 준 KDRIs 대비 적정량을 시각적인 프로그레스 바(막대그래프)로 렌더링.
   - 가격 부담으로 특정 영양제를 목록에서 제외하면, 클라이언트 단에서 교차 분석을 통해 **줄어든 영양소의 막대가 실시간으로 감소하는 연동 애니메이션 로직** 자체 구현.
3. **Clinical Interactions (상호작용 검증 UI)**: 
   - AI가 반환한 데이터를 바탕으로 **시너지(Positive), 주의(Negative), 복용 시간(Timing)**의 3가지 유형으로 자동 분류하여 사용자에게 시각적 경고 카드 제공.

---

## 🚧 5. 프로젝트의 한계점 및 향후 개선 방향
- **비용 및 응답성 (Latency)**: LLM 매 주기 호출 시 복잡한 추론 비용과 다소 긴 대기시간(수 초)이 소요됩니다. (향후 유사 증상 조합 결과를 Vector DB 등에 캐싱해 두어 즉시 리턴하는 아키텍처 개선 시급)
- **아키텍처 및 보안 한계**: 구조상 API 키 노출 등의 위험과 CORS 등을 처리하기엔 프론트엔드 직접 호출은 부적합합니다. 차후 서비스 스케일업 시 Python(FastAPI)이나 Node 백엔드를 두고 LangChain/LangGraph 프레임워크로 엔드포인트를 이전해야 합니다.
- **환각(Hallucination) 관리**: AI가 DB에 존재하지 않는 성분을 혼동해 분석 지표에 노출할 리스크가 미세하게 남아 있습니다. 추천된 영양소명과 `products.json` 간의 클라이언트 단 텍스트 매치 검증 알고리즘을 도입하여 리스크를 크게 줄였습니다.
